# 🔍 Entity Group Breakdown

## 1. **Identity Core**
- identity_record  
- identity_provenance  
- consent_record  

These define the **legal identity**, its **origin**, and **user rights**.

---

## 2. **Biometric Layer**
- biometric_capture_event  
- biometric_template  
- face_template  
- fingerprint_template  
- iris_template  
- retina_template  
- biometric_revocation_record  

This is the **multi‑modal biometric spine** of the system.

---

## 3. **FRT Layer**
- frt_query  
- frt_match_event  

This governs **authorized**, **logged**, **purpose‑limited** facial recognition usage.

---

## 4. **Fraud & Synthetic Identity Analytics**
- fraud_signal  
- identity_risk_score  
- fraud_case  
- fraud_case_identity  
- fraud_case_signal  

This is the **fraud‑detection engine**, identifying:
- stolen identities  
- synthetic identities  
- multi‑ID/single‑face anomalies  
- single‑ID/multi‑face anomalies  

---

## 5. **Governance & Oversight**
- access_policy  
- access_policy_role  
- access_policy_purpose  
- audit_log_entry  

This enforces:
- legality  
- accountability  
- transparency  
- anti‑abuse protections  

---

# 🔗 Relationship Summary

### Identity ↔ Biometrics
- One identity → many biometric capture events  
- One capture event → many biometric templates  
- One template → one modality‑specific template  

### Identity ↔ Fraud Analytics
- One identity → many fraud signals  
- One identity → many risk scores  
- Many identities ↔ many fraud cases  

### Identity ↔ FRT
- FRT match events link identity + template + query  

### Governance
- Every action → audit_log_entry  
- Every audit_log_entry → access_policy  

---

# 🧠 Cross‑System Flow (How the whole system behaves)

### 1. Enrollment
IdentityRecord → CaptureEvent → BiometricTemplate → ModalityTemplate

### 2. FRT Use
FRTQuery → FRTMatchEvent → AuditLogEntry

### 3. Fraud Detection
FraudSignal → IdentityRiskScore → FraudCase → AuditLogEntry

### 4. Governance
AccessPolicy → AuditLogEntry → Enforcement
